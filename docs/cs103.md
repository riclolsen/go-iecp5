# `cs103` — IEC 60870-5-103 master over serial (FT1.2)

The `cs103` package implements the **primary station (master)** side of
IEC 60870-5-103, the companion standard for the informative interface of
protection equipment (protective relays). The link layer is the same FT1.2
unbalanced procedure as IEC 60870-5-101 (the package reuses the `cs101`
frame codec); the application layer is 103-specific:

```
ASDU = | type ID | VSQ | cause | common address | FUN | INF | elements... |
bytes |    1    |  1  |   1   |       1        |  1  |  1  |      n      |
```

Key differences from 101/104:

- All identifier fields are **one octet** — there is no `asdu.Params`.
- Information objects are addressed by **function type (FUN)** and
  **information number (INF)** instead of an information object address.
  Standard FUNs: `FunDistanceProtection` (128), `FunOvercurrentProtection`
  (160), `FunTransformerDiff` (176), `FunLineDiff` (192), `FunGeneric`
  (254), `FunGlobal` (255). Common INFs are provided as `Inf*` constants
  (trips, starts, zones, supervision, measurand groups, ...).
- Events carry the 4-octet `CP32Time2a` time (ms/min/hour; the date comes
  from the local clock on decode); time synchronization uses CP56Time2a.
- Measurands are signed 13-bit values with OV/ER flags
  (`Measurand.Float64()` gives the fraction of full scale; rated value is
  1/1.2 or 1/2.4 of full scale depending on device parameterization).
- The link address and the ASDU common address are conventionally equal;
  the package uses one address per device for both.

## Usage

```go
option := cs103.NewOption()
cfg := cs103.DefaultConfig()
cfg.Serial = cs103.SerialConfig{ // alias of cs101.SerialConfig
	Address:  "COM3",
	BaudRate: 9600,
	DataBits: 8,
	StopBits: serial.OneStopBit,
	Parity:   serial.EvenParity, // 8E1 is the 103 standard framing
}
cfg.LinkAddress = 3 // device address
_ = option.SetConfig(cfg)
// multi-drop: option.AddSecondaryAddress(3); option.AddSecondaryAddress(4); ...

client := cs103.NewClient(handler, option) // nil if serial config missing
client.SetOnDeviceActiveHandler(func(c *cs103.Client, addr byte) {
	log.Printf("device %d link active", addr)
})
_ = client.Start()
// ...
_ = client.Close()
```

The client runs the whole 103 procedure automatically:

1. Request status of link → reset of the communication unit → link active.
   The device then reports its **identification message** (ASDU 5) which is
   collected via a class 1 poll and delivered to `IdentificationHandler`.
2. With `Config.AutoInit` (default true), every newly active device gets a
   **time synchronization** (ASDU 6) followed by a **general interrogation**
   (ASDU 7) automatically.
3. Continuous **class 2 polling** collects cyclic measurands; a set ACD bit
   switches to class 1 to fetch pending events. FCB tracking, duplicate
   suppression, T1/T2 retries and multi-drop round-robin work as in cs101.

Requests and commands (all queued, transmitted when the link is free):

```go
_ = client.TimeSync(3)                       // ASDU 6 with time.Now()
_ = client.GeneralInterrogation(3, 1)        // ASDU 7, scan number 1
_ = client.GeneralCommand(3,                 // ASDU 20
	cs103.FunOvercurrentProtection, cs103.InfAutoRecloserActive,
	cs103.DCOOn, 42 /* RII */)
_ = client.SendResetCU(3)                    // link-layer reset (FC 0)
_ = client.SendResetFCB(3)                   // 103-specific reset (FC 7)
// raw/custom ASDUs:
_ = client.SendTo(cs103.NewASDU(...), 3)
```

The command acknowledgement returns as **ASDU 1 with cause 20 (positive) or
21 (negative)**, delivered to `TimeTaggedHandler`; the `Sin` field carries
the RII of the original command.

## Handler interface

```go
type ClientHandlerInterface interface {
	TimeTaggedHandler(*ASDU, TimeTaggedInfo) error     // ASDU 1/2: events, GI replies, command acks
	MeasurandsHandler(*ASDU, MeasurandsInfo) error     // ASDU 3/9: measurands
	IdentificationHandler(*ASDU, IdentificationInfo) error // ASDU 5
	GITerminationHandler(*ASDU, byte) error            // ASDU 8: scan number
	ASDUHandler(*ASDU) error                           // ASDU 4, 6, generic, disturbance...
	ASDUHandlerAll(*ASDU) error                        // sees every ASDU first
}
```

The producing device is identified by `a.CommonAddr`. The cause of
transmission distinguishes what an ASDU 1 means: `CauseSpontaneous` (event),
`CauseGI` (interrogation reply), `CauseCommandAckPos`/`CauseCommandAckNeg`
(command result), `CauseResetCU`/`CauseStartRestart`/`CausePowerOn`
(device restart indications).

Typed decoding for other types is available on the ASDU itself:
`GetTimeTagged`, `GetMeasurands`, `GetTimeTaggedMeasurands` (ASDU 4,
short-circuit location), `GetIdentification`, `GetTimeSync`,
`GetGITermination`, `GetGeneralCommand`. All getters return errors on
malformed input (they never panic).

## Configuration (`cs103.Config`)

| Field | Range | Default | Meaning |
|-------|-------|---------|---------|
| `Serial` | — | must be set | serial port parameters (8E1 typical) |
| `LinkAddress` | 0–254 | 1 | default device address (255 = broadcast) |
| `TimeoutResponseT1` | 1–255 s | 10 s | response timeout for confirmed frames |
| `TimeoutRepeatT2` | 1–255 s, < T1 | 5 s | retry timeout after the first repetition |
| `TimeoutTestT3` | 1 s–48 h | 20 s | idle time before a keep-alive class 2 poll |
| `TimeoutSendLinkMsg` | 1 ms–10 s | 200 ms | poll/transmit scheduler pacing |
| `MaxSendQueueSize` | > 0 | 100 | outbound ASDU queue capacity |
| `AutoInit` | — | true | automatic time sync + GI on link activation |

## Not implemented

- Generic services (ASDU 10/11/21 — generic data/identification/command):
  the type identifiers and raw send/receive work (`ASDUHandler` +
  `NewASDU`/`AppendBytes`), but there are no structured GIN/GDD/GID codecs.
- Disturbance data transfer (ASDU 23–31) — the 103 equivalent of file
  transfer.
- The secondary (device) side — this package is a master only.
