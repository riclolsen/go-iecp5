# `cs101` — IEC 60870-5-101 over serial (FT1.2)

The `cs101` package implements the IEC 60870-5-101 link layer (FT1.2 frame
format, IEC 60870-5-1/-2) over a serial port, in both transmission modes:

- **Unbalanced** (default): one primary station (master) polls one or more
  secondary stations (slaves/outstations) on a shared line. Only the
  primary initiates transmissions; secondaries answer polls.
- **Balanced** (point-to-point): both stations may transmit spontaneously;
  each runs a primary role (sending) and a secondary role (acknowledging).

Types:

- `cs101.Client` — the **primary station** (master).
- `cs101.Server` — the **secondary station** (slave/outstation).

Default ASDU parameters: `asdu.ParamsStandard101` (COT 1, CA 1, IOA 2).
Set the same `asdu.Params` on both ends (`SetParams`), and give both ends
the same link-layer settings (`LinkAddrSize`, addresses, mode).

## Serial configuration

```go
serialCfg := cs101.SerialConfig{
	Address:  "COM3",            // or "/dev/ttyUSB0"
	BaudRate: 9600,
	DataBits: 8,
	StopBits: serial.OneStopBit, // go.bug.st/serial constants
	Parity:   serial.EvenParity, // 8E1 is the IEC 101 standard framing
	Timeout:  5 * time.Second,   // read timeout; 0 = block forever
}
```

## TCP-encapsulated 101 (terminal servers / serial-device servers)

The FT1.2 frames can be carried over a TCP stream instead of a local serial
port — the common setup with terminal servers, serial-device servers or
gateway software. Select the transport in the config; everything else
(link procedure, handlers, ASDUs) is identical:

```go
cfg := cs101.DefaultConfig()
cfg.Transport = cs101.TransportTCPClient        // dial out to the endpoint
cfg.TCP = cs101.TCPConfig{
	Address:        "10.0.0.9:2400",
	ConnectTimeout: 10 * time.Second,           // default 30 s
	// TLSConfig: &tls.Config{...},             // optional TLS
}
// cfg.Serial is not used with the TCP transports
```

- `TransportTCPClient` dials `TCP.Address` (works for both `Client` and
  `Server` — whichever side initiates the TCP connection).
- `TransportTCPServer` listens on `TCP.Address` and serves **one incoming
  connection at a time**; on disconnection it accepts the next one. Set the
  retry pacing on the outstation with `Server.SetReconnectInterval`.
- `TLSConfig` (both modes) wraps the stream in TLS.

Note the semantics are unchanged: this is still 101 (FT1.2 link layer with
FCB/ACK and polling) inside a TCP pipe — not IEC 104. Keep T1/T2 generous
enough for the added network latency.

## Link configuration (`cs101.Config`)

| Field | Range | Default | Meaning |
|-------|-------|---------|---------|
| `Transport` | serial / tcp-client / tcp-server | serial | how the FT1.2 frames are carried |
| `Serial` | — | must be set (serial) | serial port parameters |
| `TCP` | — | must be set (TCP) | TCP address, connect timeout, optional TLS |
| `Mode` | `ModeUnbalanced` / `ModeBalanced` | unbalanced | transmission procedure |
| `LinkAddress` | fits `LinkAddrSize` | 1 | this station's address (server) / target secondary (client) |
| `LinkAddrSize` | 1–2 | 1 | link address octets on the wire (must match both ends) |
| `TimeoutResponseT1` | 1–255 s | 10 s | response timeout for confirmed frames |
| `TimeoutRepeatT2` | 1–255 s, < T1 | 5 s | retry timeout after the first repetition |
| `TimeoutTestT3` | 1 s–48 h | 20 s | idle time before a Test-Link keep-alive |
| `TimeoutSendLinkMsg` | 1 ms–10 s | 200 ms | pacing of the poll/transmit scheduler |
| `MaxSendQueueSize` | > 0 | 100 | outbound ASDU queue / class buffer capacity |

`TimeoutSendLinkMsg` is effectively the **polling period** of the unbalanced
master: every tick it either transmits queued user data or polls the next
secondary for class 1/2 data. Lower it for lower latency, raise it to reduce
line traffic.

## Primary station (master) — `cs101.Client`

```go
option := cs101.NewOption()
cfg := cs101.DefaultConfig()
cfg.Serial = serialCfg
cfg.LinkAddress = 1                 // secondary station to poll
_ = option.SetConfig(cfg)
_ = option.SetParams(asdu.ParamsStandard101)
option.SetAutoReconnect(true).SetReconnectInterval(30 * time.Second)

client := cs101.NewClient(handler, option) // handler: cs101.ClientHandlerInterface; nil on bad serial config
client.SetOnConnectHandler(func(c *cs101.Client) { ... })       // first frame received
client.SetConnectionLostHandler(func(c *cs101.Client, err error) { ... })
client.SetConnectErrorHandler(func(c *cs101.Client, err error) { ... })
_ = client.Start()   // opens the port in the background
// ...
_ = client.Close()
```

The client runs the link procedure automatically — you do not have to drive
it:

1. Request Status of Link → 2. Reset of Remote Link → 3. link active
   (`client.IsLinkActive()` becomes true).
4. Unbalanced: continuous class 2 polling; a set ACD bit in any response
   triggers a class 1 request. Queued user data (commands) is interleaved.
5. Frame Count Bit (FCB) is tracked per station across all FCV frames and
   repeated frames are retransmitted with the same FCB, per IEC 60870-5-2.
6. Response timeout: one retry after T1 (then T2); on total failure the
   station is re-initialized. With a single secondary (or balanced mode)
   the connection is torn down and re-established (auto-reconnect).

Sending commands (queued, transmitted when the link is free):

```go
_ = client.InterrogationCmd(asdu.CauseOfTransmission{Cause: asdu.Activation}, 1, asdu.QOIStation)
_ = client.ClockSynchronizationCmd(asdu.CauseOfTransmission{Cause: asdu.Activation}, 1, time.Now())
// also: CounterInterrogationCmd, ReadCmd, ResetProcessCmd, DelayAcquireCommand, TestCommand

// control commands via the asdu helpers:
_ = asdu.SingleCmd(client, asdu.C_SC_NA_1,
	asdu.CauseOfTransmission{Cause: asdu.Activation}, 1,
	asdu.SingleCommandInfo{Ioa: 6000, Value: true})

// target a specific secondary on a multi-drop line:
_ = client.SendTo(pack, 5)
```

Manual link-layer actions (rarely needed; executed by the protocol loop when
the link is free): `SendResetLink()`, `SendRequestLinkStatus()`,
`SendReqDataClass2()`.

### Multi-drop (several secondaries on one line)

```go
option.AddSecondaryAddress(1)
option.AddSecondaryAddress(2)
option.AddSecondaryAddress(3)
```

The client initializes and polls all stations round-robin, keeps FCB/link
state per station, and continues serving healthy stations when one dies.
`client.Send(pack)` targets the first configured address; use
`client.SendTo(pack, addr)` for the others. The ASDU **common address** and
the **link address** are distinct concepts — map them in your application
(they are often configured equal in the field).

### ClientHandlerInterface

```go
type ClientHandlerInterface interface {
	InterrogationHandler(*asdu.ASDU) error        // interrogation responses (COT 20/21..36)
	CounterInterrogationHandler(*asdu.ASDU) error // counter responses (COT 37..41)
	ReadHandler(*asdu.ASDU) error
	TestCommandHandler(*asdu.ASDU) error
	ClockSyncHandler(*asdu.ASDU) error
	ResetProcessHandler(*asdu.ASDU) error
	DelayAcquisitionHandler(*asdu.ASDU) error
	ASDUHandler(*asdu.ASDU, int) error            // everything else (int = client number)
	ASDUHandlerAll(*asdu.ASDU, int) error         // sees every ASDU first
}
```

Unlike cs104, the cs101 client routes by **cause of transmission**:
interrogation-caused M_* data goes to `InterrogationHandler`,
counter-request-caused M_IT data to `CounterInterrogationHandler`, and
everything else (spontaneous data, command confirmations, end of init) to
`ASDUHandler`. Decode with the `Get*` getters as usual.

## Secondary station (slave) — `cs101.Server`

```go
srv := cs101.NewServer(handler) // handler: cs101.ServerHandlerInterface
cfg := cs101.DefaultConfig()
cfg.Serial = serialCfg
cfg.LinkAddress = 1             // this station's link address
srv.SetConfig(cfg)
srv.SetParams(asdu.ParamsStandard101)
_ = srv.Start()                 // opens the port in the background
// ...
_ = srv.Close()
```

The server answers the link procedure automatically (status, reset, test,
FCB duplicate detection with retransmission of the last response) and
dispatches received ASDUs to the handler.

### Sending data from the secondary

`srv.Send(pack)` — and the `asdu.*` helpers with `srv` as the connection —
**buffer** the ASDU; nothing is transmitted spontaneously in unbalanced mode
(the standard forbids it). Delivery happens when the master polls:

- Causes `Periodic`/`Background` → **class 2 buffer** (cyclic data).
- Every other cause (spontaneous, interrogation responses, confirmations)
  → **class 1 buffer** (events).
- Pending class 1 data is announced with the **ACD bit** on every response,
  which makes a standard master issue a class 1 poll.
- An empty poll is answered with "requested data not available" (FC 9).
- A full buffer makes `Send` return `cs101.ErrSendQueueFull`.

```go
// e.g. inside InterrogationHandler:
_ = pack.SendReplyMirror(srv, asdu.ActivationCon)
_ = asdu.Single(srv, false, asdu.CauseOfTransmission{Cause: asdu.InterrogatedByStation},
	pack.CommonAddr, points...)
_ = pack.SendReplyMirror(srv, asdu.ActivationTerm)
```

### ServerHandlerInterface

```go
type ServerHandlerInterface interface {
	InterrogationHandler(*asdu.ASDU, asdu.QualifierOfInterrogation) error
	CounterInterrogationHandler(*asdu.ASDU, asdu.QualifierCountCall) error
	ReadHandler(*asdu.ASDU, asdu.InfoObjAddr) error
	ClockSyncHandler(*asdu.ASDU, time.Time) error
	ResetProcessHandler(*asdu.ASDU, asdu.QualifierOfResetProcessCmd) error
	DelayAcquisitionHandler(*asdu.ASDU, uint16) error
	ASDUHandler(*asdu.ASDU) error          // everything else, incl. control commands
	ASDUHandlerAll(*asdu.ASDU, int) error  // sees every ASDU first
}
```

The `*asdu.ASDU` passed to the handlers carries the full identifier and
information-object payload; the qualifier/time/IOA you usually need is
additionally extracted into the handler's dedicated parameters. For control
commands arriving at `ASDUHandler`, switch on `pack.Type`, decode with the
`Get*` getters, and reply with `pack.SendReplyMirror(srv, ...)`; buffer
return information with the `asdu.*` send helpers.

## Balanced mode (point-to-point)

Set `cfg.Mode = cs101.ModeBalanced` on **both** stations and give both the
**same** `LinkAddress`. Use `cs101.Client` on station A (the DIR bit is set
on its frames) and `cs101.Server` on station B.

- Both stations initialize their transmit direction with Reset of Remote
  Link and may then send confirmed user data spontaneously.
- `client.Send`/`srv.Send` transmit directly (no class polling; the class
  buffers are not used).
- Each side acknowledges the peer's frames and performs FCB duplicate
  detection; response timeouts retry once and then re-initialize the
  direction.

## Behavior and troubleshooting

- **FT1.2 framing**: fixed-length frames (link commands/confirms), variable
  length frames (`68 L L 68 ... cs 16`) for ASDUs, single char `E5` accepted
  as positive ack. Checksum or length violations are logged and skipped;
  they never crash the receiver.
- The primary's `onConnect` callback fires on the **first received frame**
  (i.e. when the line is actually alive), not when the port opens.
- The link address on the wire always identifies the **secondary** station
  (in both directions), per IEC 60870-5-2.
- If the master logs `unexpected link address`, `LinkAddress`/`LinkAddrSize`
  disagree between the ends.
- If ASDUs decode as garbage (wrong type/cause), the `asdu.Params` differ
  between the ends.

## Errors

| Error | Meaning |
|-------|---------|
| `cs101.ErrUseClosedConnection` | serial port not open |
| `cs101.ErrNotActive` | link not initialized yet |
| `cs101.ErrSendQueueFull` | outbound queue / class buffer full |
| `cs101.ErrBufferFulled` | internal frame channel full |
| `cs101.ErrTimeoutT1` | response timeout after retry (connection re-established) |
